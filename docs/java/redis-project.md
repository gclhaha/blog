# Redis 项目实战总结

本文总结一些面试中经常问到的Redis相关问题，并通过项目中的实际使用进行总结。

## 项目中有那些业务使用了Redis，都用来做什么？

1. 缓存权限，权限在系统中频繁使用，且不会频繁变动。缓存数据，分散数据库压力。
2. 业务加锁，在订单创建、库存扣减通过商户id、订单号或者商品id进行加锁，防止重复提交订单或者库存超卖。过期时间30秒，此为高负载情况下业务最长处理时间，过期后自动释放锁。

## 如果Redis过期了，但是业务还没有结束，是怎么处理的？

项目中引入了Redssion，订单业务锁定后，如果业务执行超时，Redssion则会通过看门狗机制，来进行锁的续期。

## 什么是缓存穿透、缓存击穿、缓存雪崩？在项目中是怎么处理的？

### 缓存穿透

缓存穿透，是攻击者查询一个根本不存在的数据，由于查询的内容，既不在缓存，也不在数据库，所以每次都会去查询数据库，导致数据库瘫痪。

处理方式：在接口入口校验参数合法。创建一个空值的缓存值，并设置较短的时间，但是这个方案可能还是会因为攻击者一直使用随机参数导致大量空值的数据。布隆过滤器，系统启动后+定时任务，将可能存在的key加载布隆过滤器，请求先通过过滤器，如果不存在直接返回错误，如果key存在，继续走缓存和数据库。单机可以使用Guava库的RedisBloom，集群可以使用Redisson的RBloomFilter。

### 缓存击穿

缓存击穿，是某个高请求量的key过期了，此时大量请求会访问数据库，增加数据库压力。

处理方式：使用互斥锁，当缓存失效时，不让所有请求都去查数据库，只允许获取锁的请求去查询数据库并重写缓存，其他请求等待或返回兜底数据。

### 缓存雪崩

缓存雪崩，是在短时间内，大量的缓存key过期，或者redis不可用。这时所有的请求都会走数据库，导致数据库卡死。

处理方式：1. 可以在设置key时，比如可以用random(1, 5) min 设置一个扰动值，这样可以避免所有的key都是同一个过期时间。2. 其次就是处理redis的高可用，使用redis集群，保证服务可用。3. 进阶方案就是使用二级缓存，因为redis的获取和设置，也存在网络损耗或者网络故障，可以先读本地缓存，再读redis，再读数据库。不过此方案需要封装缓存的操作，增加了一些代码成本。不过在AI时代，几句话也就能实现了。
