# 消息队列项目实战总结

## 你知道那些消息队列？他们有什么特点？

1. Kafka：在这三个队列里吞吐量最高，是一个分布式的、分区的、基于提交日志的流处理平台。充分利用了操作系统的页缓存（Page Cache）进行顺序读写，并结合零拷贝（Zero-Copy）技术，使得它在海量数据写入时依然能保持极高的效率。通过分区（Partition）和副本（Replica）机制，Kafka 可以非常容易地进行横向扩展，具备极高的数据可靠性。优点是高吞吐量、缺点是存在延迟，以及特性较少，对死信队列、延迟消息、分布式事务等高级业务功能的支持较为有限
2. RabbitMQ：是基于 Erlang 语言开发的，实现了 AMQP 协议，可以实现微秒级的极低延迟。路由机制强大，Direct、Topic、Fanout 等不同类型的交换器，它可以实现非常灵活的消息路由逻辑。提供了发送确认（Publisher Confirms）、消费者应答（ACK）、死信队列（DLX）机制，确保消息在复杂的传输路径中不轻易丢失。并且提供直接可用的可视化洁面，方便操作。缺点就是吞吐量通常在万到10万级的TPS，比较适合对于高吞吐量
3. RocketMQ：特点是原生支持分布式事务，支持延迟消息，分区消费。适合高一致性和事物要求高的业务，比如金融和电商。

## 用过哪几种消息队列？使用场景是什么？

Kafka：在这三个队列里吞吐量最高，平时业务中也用的最多。1. 主要用在业务的异步通知，比如订单创建成功，需要预扣库存或者优惠券，会员的积分、储值、消费统计，会通过消息队列异步去处理。2. 或者同步搜索引擎的索引，在用户、商品的新增、删除，会通过消息队列通知搜索服务进行更新。 3.实现统一的系统消息中心，需要发送的消息,比如会员的积分变动、储值变动、优惠券过期提醒，订单成功等，都通过消息中心组件触发给web或者小程序。4.通过消息队列，同步给其他系统用户的上下线信息

## 如何保证消息可靠？

消息可靠要考虑三个方面，一个是生产者方，一定要确保成功发送到broker。二是要将消息持久化，避免消息因为宕机、硬件故障特殊原因导致消息消失，并且消费者可以根据持久化记录来做业务处理。三是消费者要手动确认ACK，并且将消息放入死信队列，如果不支持死信队列，则可以通过持久化的消息记录更新状态，后续扫描定时执行或者手动触发补偿。

## 如果消费者出现错误，消息出现了大量积压应该怎么办？

首先要确认是否是消费者的bug导致崩溃或者重试，如果是bug引起的，首先要进行回滚恢复业务。如果没有发现bug报错，那就要查询数据库是否出现瓶颈，可以增加数据库连接数。消费者恢复后，消息积压过多消费过慢，如果是 RabbitMQ 则可以通过增加消费者数量，他们会竞争消费。如果是 Kafka 或者 RocketMQ，他们是基于分区的，一个分区只能有一个消费者。此时需要创建一个新的临时topic，并且将分区设置为原来的多倍（如果原来是3个，现在设置为30个）。此时更改原来消费者的业务，只做转发，将所有的消息写入临时的topic。部署多个临时消费者实例（或者增加消费者的线程数量），共同消费临时的topic，来变相的增加消费速度。等消费完毕后，再将消费者的业务恢复为直接消费，将临时的实例关闭。
